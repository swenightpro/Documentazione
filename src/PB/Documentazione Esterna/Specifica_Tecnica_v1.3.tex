\section{Introduzione}

\subsection{Scopo del documento}
Il presente documento ha lo scopo di descrivere l'architettura software del sistema \textit{SmartOrder}, definendo le scelte tecnologiche, i pattern architetturali e progettuali adottati, e l'architettura di dettaglio di ciascun componente.
La Specifica Tecnica costituisce il riferimento principale per la fase di codifica, garantendo coerenza tra progettazione e implementazione.

\subsection{Scopo del prodotto}
\textit{SmartOrder} \`e una piattaforma B2B per l'automazione della gestione degli ordini commerciali. Il sistema consente ai clienti di formulare ordini in linguaggio naturale (testo, audio, immagini) e li converte automaticamente in ordini strutturati pronti per l'integrazione con sistemi ERP aziendali, riducendo errori e tempi di inserimento manuale.

\subsection{Glossario}
Per evitare ambiguit\`a, i termini tecnici e di dominio utilizzati nel documento sono definiti nel \textit{Glossario v2.0}.
I termini presenti nel glossario sono contrassegnati con il simbolo \textsubscript{\textsc{g}} alla prima occorrenza in ogni sezione.

In questa sezione vengono elencate le tecnologie adottate per lo sviluppo del sistema \textit{SmartOrder}. Per ciascuna tecnologia viene indicata la versione utilizzata, la motivazione della scelta e il contesto d'impiego nel progetto.

\subsection{Linguaggi di programmazione}
% -------------------------------------------------------------------

\subsubsection{TypeScript}
\begin{itemize}[noitemsep]
\item \textbf{Versione:} 5
\item \textbf{Documentazione ufficiale:} \href{https://www.typescriptlang.org/docs/}{TypeScript
 Documentation} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Frontend
\item \textbf{Contesto di utilizzo:} linguaggio principale per l'interfaccia utente dell'applicativo Next.js, sia per i componenti React che per la gestione dello stato lato client.
\item \textbf{Vantaggi:} tipizzazione statica che riduce errori a runtime, ottimo supporto IDE, compatibilit\`a nativa con React e Next.js.
\item \textbf{Svantaggi:} overhead di compilazione rispetto a JavaScript puro.
\end{itemize}

\subsubsection{Python}
\begin{itemize}[noitemsep]
\item \textbf{Versione:} 3.11
\item \textbf{Documentazione ufficiale:} \href{https://docs.python.org/3/}{Python 3 Documentation} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Backend / Modulo AI
\item \textbf{Contesto di utilizzo:} implementazione della pipeline di elaborazione degli ordini tramite intelligenza artificiale e gestione delle API tramite FastAPI.
\item \textbf{Vantaggi:} ecosistema maturo in ambito NLP/ML, compatibilit\`a nativa con le librerie OpenAI, sintassi concisa, supporto asincrono con Uvicorn/ASGI.
\item \textbf{Svantaggi:} prestazioni inferiori a linguaggi compilati; mitigato dall'uso di Uvicorn (ASGI asincrono).
\end{itemize}

\end{itemize}

Le seguenti tabelle organizzano le dipendenze per ambito tecnico, evidenziando per ciascun elemento il ruolo operativo nel progetto, il criterio di adozione rispetto ad alternative praticabili e l'integrazione con il resto dello stack applicativo.

\subsubsection{Frontend -- Core}
\begin{table}[H]
    \centering
    \renewcommand{\arraystretch}{1.2}
    \begin{tabularx}{\textwidth}{@{}l l X@{}}

        Next.js & $\geq$ 15.5 & Framework di riferimento per il frontend applicativo: e' stato adottato per fornire una struttura unificata di routing, rendering e organizzazione dei moduli invece di una configurazione manuale basata su tool separati; si integra in modo nativo con React 19 e TypeScript 5 nel ciclo di sviluppo e build. \\
        React & $\geq$ 19 & Libreria di composizione dell'interfaccia utente: e' stata scelta per il modello a componenti e per la maturita' dell'ecosistema rispetto ad approcci meno diffusi nel contesto del progetto; costituisce il livello UI eseguito all'interno dell'applicazione Next.js. \\
        TypeScript & $\geq$ 5 & Linguaggio principale del frontend: e' stato preferito a JavaScript non tipizzato per ridurre ambiguita' contrattuali tra moduli e facilitare la manutenzione evolutiva; si integra con React e Next.js tramite tipizzazione condivisa di componenti, payload e utilita'. \\

\section{Architettura di deployment}

Questa sezione descrive la distribuzione fisica dei componenti del sistema \textit{SmartOrder}, illustrando gli ambienti di esecuzione, i nodi
     infrastrutturali coinvolti e le strategie di Continuous Integration e Continuous Deployment (CI/CD\ped{g}).

\subsection{Architettura: Sottosistemi Monolitici Indipendenti}
Il sistema \textit{SmartOrder} adotta un'architettura \textbf{Client-Server} basata su due unità di deployment indipendenti. Entrambi i componenti
      sono strutturati internamente come monoliti, garantendo una gestione semplificata del rilascio pur mantenendo una netta separazione delle
      responsabilità:

\begin{itemize}
    \item \textbf{Backend (Modular Monolith)}: Il server è un monolite modulare implementato in FastAPI (Python). Nonostante venga eseguito come un
      unico processo, la logica di business è isolata verticalmente per domini funzionali, assicurando coesione interna e facilità di manutenzione.
    \item \textbf{Frontend (Frontend Monolith)}: Il client è un'applicazione Next.js distribuita come unità singola. Gestisce interamente lo strato
      di presentazione e lo stato dell'interfaccia utente, operando in un ambiente di runtime Node.js separato e dedicato.
\end{itemize}
Il disaccoppiamento tra i due sottosistemi è garantito dall'utilizzo di interfacce HTTP per lo scambio di dati in formato JSON.
L'organizzazione del codice segue l'approccio \textbf{Monorepo}: entrambi i sottosistemi risiedono in un unico repository dei sorgenti. Questa
scelta semplifica la gestione delle dipendenze e garantisce la coerenza tra le interfacce del frontend e le API del backend durante lo sviluppo.

\subsubsection{Motivazioni della scelta architettonica}
L'adozione di questa struttura garantisce un equilibrio ottimale tra semplicità operativa e flessibilità evolutiva, offrendo i seguenti vantaggi:
\begin{itemize}
    \item \textbf{Efficienza Infrastrutturale}: L'approccio monolitico riduce drasticamente l'overhead di gestione rispetto a un'architettura a
     microservizi, pur mantenendo una chiara separazione delle responsabilità.
    \item \textbf{Scalabilità Indipendente}: La natura disaccoppiata dei due sottosistemi permette di scalare le risorse di calcolo in modo
     differenziato (es. aumentando le risorse del solo backend durante picchi di elaborazione AI).
    \item \textbf{Coerenza del Ciclo di Vita}: L'organizzazione in Monorepo assicura che il rilascio di nuove funzionalità avvenga in modo atomico,
     eliminando il rischio di disallineamento tra l'interfaccia utente e le logiche del server.
\end{itemize}

\subsection{Modularità e organizzazione dei sottosistemi}
Il sistema garantisce un elevato grado di manutenibilità grazie all'organizzazione modulare del codice, che permette di gestire la complessità dei
      due sottosistemi senza ricorrere a un'architettura a microservizi.

\subsubsection{Isolamento verticale del Backend}
Il backend è strutturato secondo i principi della \textbf{modularità verticale}. Ogni dominio di business (es. gestione ordini, carrello, ticket) è
      racchiuso in un modulo indipendente che contiene sia la logica applicativa che le interfacce per l'accesso ai dati. Questo isolamento permette di:
\begin{itemize}
    \item Evitare dipendenze circolari tra le diverse funzionalità del sistema.
    \item Facilitare il collaudo dei singoli moduli in isolamento.
    \item Semplificare l'evoluzione di una specifica parte del sistema senza impatti sulle altre.
\end{itemize}

\subsubsection{Componenti di un modulo}
Per garantire uniformità e coesione, ogni modulo del backend si articola nei seguenti componenti logici:
\begin{enumerate}
    \item \textbf{Controller}: Punto di ingresso per le richieste HTTP e validazione dell'input.
    \item \textbf{Service}: Implementazione della logica di business e delle regole di dominio.
    \item \textbf{Port}: Definizione astratta dei requisiti infrastrutturali (es. interfacce per il database).
    \item \textbf{Schema}: Modelli di dati utilizzati per lo scambio di informazioni (DTO).
    \item \textbf{Adapter}: Implementazione specifica delle Port per le tecnologie utilizzate (es. PostgreSQL, OpenAI).
\end{enumerate}

      sorgente.

\subsection{Canali di comunicazione}
Le interazioni tra i diversi componenti del sistema avvengono attraverso protocolli standardizzati, garantendo sicurezza e interoperabilità:
\begin{itemize}
    \item \textbf{Frontend $\leftrightarrow$ Backend}: La comunicazione avviene tramite chiamate API su protocollo cifrato HTTPS, con scambio

    \item \textbf{Backend $\leftrightarrow$ PostgreSQL}: Il backend comunica con il database tramite protocollo TCP, ottimizzato attraverso un

    \item \textbf{Backend $\leftrightarrow$ OpenAI API}: Le richieste ai servizi di intelligenza artificiale vengono effettuate tramite invocazioni

    \item \textbf{Backend $\rightarrow$ File Storage}: L'integrazione con il sistema ERP avviene tramite operazioni di I/O su filesystem per il
      deposito dei file d'ordine in formato JSON.
\end{itemize}

\subsection{Aspetti operativi e Continuous Deployment}
Il ciclo di vita del software è gestito tramite pratiche automatizzate di \textbf{Continuous Deployment} (CD\ped{g}) per assicurare rilasci
frequenti, coordinati e privi di disservizi.
\begin{itemize}
    \item \textbf{Rilascio Unificato e Automatizzato}: Grazie all'adozione di un'architettura \textbf{Monorepo}, il processo di deployment è
     centralizzato. La piattaforma \textbf{Railway} è interfacciata direttamente con il repository GitHub e, ad ogni aggiornamento sul branch principale,
     avvia simultaneamente la build e il rilascio di entrambi i container (frontend e backend). Questo garantisce che le versioni dei due sottosistemi
     siano sempre sincronizzate e compatibili tra loro.
    \item \textbf{Aggiornamenti Zero-Downtime}: Il deployment avviene senza interruzioni di servizio per l'utente finale. I nuovi container vengono
     avviati e sostituiscono quelli precedenti solo dopo aver superato con successo i controlli di integrità del sistema, assicurando la massima
     disponibilità della piattaforma.
    \item \textbf{Gestione delle Configurazioni}: I dati sensibili, quali chiavi API e credenziali di accesso al database, non sono memorizzati nel
     codice sorgente. Vengono iniettati nei runtime come variabili d'ambiente criptate attraverso la dashboard di gestione della piattaforma cloud,
     rispettando i moderni standard di sicurezza.
\end{itemize}

\subsection{Sicurezza e protezione dei dati}
Nel progetto la sicurezza e' stata considerata in modo continuo, integrandola nelle principali scelte architetturali. Di seguito sono riportate le misure adottate:
\begin{itemize}
    \item \textbf{Difesa in profondita' contro il furto di sessione}: autenticazione basata su cookie \texttt{HTTPOnly} e \texttt{Secure}, preferiti allo storage nel browser per ridurre la superficie d'attacco XSS.
    \item \textbf{Protezione credenziali con scrypt + pepper}: le password sono salvate con salt univoco, cost factor elevato e pepper lato server, cosi' da rendere non riutilizzabili gli hash anche in caso di compromissione del solo database.
    \item \textbf{Controllo accessi real-time su SSE}: l'apertura dei canali Server-Sent Events e' subordinata a validazione JWT; gli stream sono segregati per sessione e ruolo applicativo (cliente/operatore).
    \item \textbf{Mitigazione nativa SQL Injection}: l'accesso dati e' concentrato nei repository/adapters PostgreSQL con query parametrizzate, evitando concatenazioni manuali di input utente in statement SQL.
    \item \textbf{Policy CORS restrittiva}: il backend accetta richieste solo da origini autorizzate tramite regole esplicite (incluse validazioni regex per ambienti certificati), riducendo rischi CSRF e chiamate cross-origin abusive.
\end{itemize}

Queste scelte permettono di proteggere l'intero ciclo operativo, dalla conversazione in chat alla persistenza dell'ordine e alla sua esportazione verso ERP.

\end{figure}

\subsubsection{Realizzazione}
Il Core opera operando esclusivamente su interfacce e oggetti di dominio, senza dipendere direttamente dai dettagli dei framework. Per dimostrare l'uso rigoroso del pattern Ports and Adapters, vengono applicati i seguenti principi ai confini del sistema:
\begin{itemize}
    \item \textbf{Adattamento in ingresso:} i controller FastAPI agiscono da \textit{driving adapters}. Ricevute le richieste esterne, il Controller non passa la rappresentazione grezza (es. il payload JSON in arrivo) direttamente al Service. Al contrario, valida e adatta l'input in interfacce di \textit{Use Case} (DTO o oggetti applicativi) immediatamente fruibili dalla logica di business.
    \item \textbf{Adattamento in uscita:} il Service contiene la pura logica di dominio. Quando necessita di interfacciarsi con il database o servizi esterni, passa esclusivamente oggetti di dominio alle porte sottostanti (es. interfacce dei repository), senza dipendere in alcun modo da rappresentazioni esterne, query SQL o librerie tecnologiche. Gli adapter infrastrutturali (es. i driver per PostgreSQL o OpenAI) implementano queste porte ricevendo e convertendo l'oggetto di dominio nel formato tecnico richiesto.
\end{itemize}
Questa struttura svincola del tutto il Core dalla tecnologia, assicurando un alto grado di testabilità in isolamento.

\subsection{Frontend}

\subsubsection{Esigenze Architetturali}
Il frontend richiede una separazione netta tra presentazione, stato applicativo e accesso ai dati, con supporto ad aggiornamenti real-time e interazioni frequenti sul carrello senza degradare la reattivita' dell'interfaccia.

\subsubsection{Soluzione Architetturale e Separazione delle Responsabilità}
Il frontend adotta una \textbf{Component-Based Architecture} organizzata applicando rigorosamente una scomposizione a livelli delle responsabilità:
\begin{itemize}
    \item \textbf{Presentation Logic}: I componenti puri (\textit{dumb components} all'interno di \texttt{components/ui/}) ricevono dati dai livelli superiori e gestiscono il rendering dell'interfaccia utente. Non prendono decisioni di business.
    \item \textbf{View Logic}: Componenti funzionali (\textit{smart components} o pages in \texttt{app/}) che formattano i dati, gestiscono lo stato dell'interazione locale e coordinano la renderizzazione.
    \item \textbf{Application Logic (State e Orchestrazione)}: Implementata nei custom hook (\texttt{hooks/}) e nei context provider (\texttt{contexts/} come \texttt{CartContext}). Questi orchestrano eventi asincroni e mantengono coerente lo stato locale del sistema.
    \item \textbf{Business Logic / API Abstraction Layer}: Racchiusa all'interno dei file di servizio in \texttt{services/} (es. \texttt{cartService}, \texttt{authService}). Questi agiscono come \textit{Facade}, mappando modelli di dominio ai relativi endpoint.
\end{itemize}

I pattern principali adottati nel disaccoppiamento di tali logiche includono:
\begin{itemize}
    \item \textbf{Facade}: il layer \texttt{services/} incapsula la comunicazione HTTP, isolando i componenti UI dai dettagli di rete.
    \item \textbf{Observer}: Context API e flussi SSE propagano gli aggiornamenti di stato ai componenti sottoscritti.
    \item \textbf{Singleton (module-level)}: i servizi API sono condivisi come istanze di modulo, con configurazione centralizzata.
\end{itemize}

\subsubsection{Realizzazione}
Per la gestione dello stato globale è stata adottata la \textbf{React Context API}, con separazione in due provider principali e un service layer dedicato:
\begin{itemize}
    \item \textbf{SessionContext}: gestisce identita' utente (\texttt{sessionId}), cronologia dei messaggi e ciclo di vita della sessione; integra l'hook \texttt{useSSE} per ricevere aggiornamenti real-time (es. messaggi operatore) e sincronizzarli nello stato condiviso.
    \item \textbf{CartContext}: gestisce lo stato del carrello con strategia \textit{Optimistic UI}; le modifiche alle quantita' vengono rese subito in interfaccia e inviate al backend con \textit{debounce} a 600ms, per ridurre burst di chiamate su interazioni rapide.
    \item \textbf{Servizi disaccoppiati}: la comunicazione API e' incapsulata in \texttt{services/}, cosi' i context restano focalizzati su stato, reattivita' e sincronizzazione.
\end{itemize}

Questa soluzione e' stata scelta per mantenere un equilibrio tra leggerezza e robustezza: semplicit\`a implementativa coerente con un MVP, esperienza utente fluida grazie agli aggiornamenti ottimistici, propagazione immediata degli eventi server via SSE e maggiore modularita' nel testing (componenti dipendenti da sessione e carrello testabili in modo indipendente).

\subsection{Design pattern adottati}

Questa sottosezione mantiene una vista problem-oriented, separando i pattern effettivamente adottati nel backend e nel frontend.

\subsubsection{Pattern backend}

\paragraph{Dependency Injection}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} I servizi applicativi devono essere testabili e sostituibili senza modificare controller e logica di dominio. Creare dipendenze "in linea" nei componenti avrebbe introdotto accoppiamento forte e inizializzazioni duplicate.
    \item \textbf{Soluzione Architetturale:} Le dipendenze vengono composte nel punto di avvio e iniettate nei componenti tramite provider del framework, mantenendo separati il wiring infrastrutturale e il comportamento di dominio.
    \item \textbf{Realizzazione:} Controller e servizi ricevono istanze gia' risolte; in collaudo e test si possono iniettare implementazioni alternative (mock/stub) senza intervenire sul codice applicativo.
\end{itemize}

\paragraph{Principi SOLID rispettati}
L'iniezione delle dipendenze applica il \textbf{Dependency Inversion Principle}: i servizi dipendono da astrazioni e non da classi concrete. Inoltre favorisce l'\textbf{Open/Closed Principle}, poiche' nuove implementazioni possono essere introdotte nel composition root senza modificare la logica di dominio esistente.

\noindent\textbf{Esempio:} il composition root in \texttt{main.py} costruisce adapter e broadcaster, quindi inietta le dipendenze nei servizi applicativi (\texttt{CartService}, \texttt{AuthService}, \texttt{ConversationService}). Nei test le dipendenze possono essere sostituite (es. \texttt{FakeSessionManager}) senza modificare la logica di dominio.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.80\textwidth]{componenti_specifiche_tecniche/architetturalogica/di_uml.png}
    \caption{Pattern Dependency Injection: composizione nel bootstrap e iniezione nei servizi applicativi.}
    \label{fig:di-architettura-logica}
\end{figure}

\paragraph{Repository}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} La logica di business non deve dipendere da query SQL, transazioni e dettagli di schema, per evitare propagazione delle modifiche infrastrutturali nel Core.
    \item \textbf{Soluzione Architetturale:} L'accesso ai dati e' mediato da interfacce repository orientate ai casi d'uso (lettura/scrittura ordini, sessioni, ticket), cosi' da mantenere stabile il contratto visto dal dominio.
    \item \textbf{Realizzazione:} I servizi lavorano su repository astratti; gli adapter concreti gestiscono SQL, mapping e vincoli di persistenza, preservando il disaccoppiamento tra dominio e DBMS.
\end{itemize}

\paragraph{Principi SOLID rispettati}
Il pattern supporta il \textbf{Single Responsibility Principle}: la logica applicativa resta distinta dalla persistenza. Rafforza anche il \textbf{Dependency Inversion Principle}, perche' i service dipendono dal contratto repository; il dettaglio tecnologico (PostgreSQL, mock) e' confinato negli adapter concreti.

\noindent\textbf{Esempio:} \texttt{AuthService} dipende da \texttt{IUserRepository} e usa il contratto per login/lookup utenze. L'implementazione puo' essere \texttt{PostgresAdapter} in esercizio o \texttt{FakeUserRepository} in test: il servizio mantiene invariata l'interfaccia d'uso e resta indipendente dalla persistenza.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.76\textwidth]{componenti_specifiche_tecniche/architetturalogica/repo_uml.png}
    \caption{Pattern Repository: \texttt{AuthService} dipende da \texttt{IUserRepository}, con implementazioni sostituibili.}
    \label{fig:repository-architettura-logica}
\end{figure}

\paragraph{Adapter (Ports and Adapters)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Il Core deve integrare database e servizi AI senza conoscere SDK, payload e protocolli specifici dei provider.
    \item \textbf{Soluzione Architetturale:} Le Ports definiscono contratti stabili; i driven adapter traducono le operazioni di dominio in chiamate tecniche verso le integrazioni esterne.
    \item \textbf{Realizzazione:} Nel caso AI, \texttt{ConversationService} dipende da \texttt{IAIClient} e non dall'SDK OpenAI. \texttt{OpenAIAdapter} implementa l'interfaccia e converte i metodi applicativi (\texttt{interpret\_intent}, \texttt{transcribe\_audio}, \texttt{generate\_embedding}) nelle chiamate specifiche di \texttt{AsyncOpenAI}.
\end{itemize}

\paragraph{Principi SOLID rispettati}
L'Adapter concretizza il \textbf{Dependency Inversion Principle} mantenendo stabile il contratto verso il dominio. In ottica \textbf{Open/Closed Principle}, il sistema puo' integrare nuovi provider o protocolli aggiungendo adapter dedicati senza modificare i servizi core.

\noindent\textbf{Esempio:} il diagramma mostra un adattamento \textit{port-to-sdk}: il contratto applicativo resta stabile nel Core, mentre la traduzione dei payload e delle API del provider e' confinata nell'adapter. In questo modo, un cambio di provider AI impatta il layer di integrazione senza modificare il servizio di dominio.

    \label{fig:adapter-architettura-logica}
\end{figure}

\subsubsection{Pattern frontend}

\paragraph{Facade (Service Layer frontend)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} La UI deve restare disaccoppiata da protocollo HTTP, costruzione URL, header di sicurezza, cookie CSRF e gestione degli errori applicativi.
    \item \textbf{Soluzione Architetturale:} Il Service Layer in \texttt{services/} espone servizi specialistici (es. \texttt{cartService}, \texttt{authService}, \texttt{orderService}) che agiscono da Facade verso le API backend.
    \item \textbf{Realizzazione:} I componenti React invocano operazioni di dominio ad alto livello; internamente i servizi orchestrano utilita' di basso livello (\texttt{apiFetch}, \texttt{buildApiUrl}, \texttt{getApiBaseUrl}, \texttt{getApiErrorMessage}) e uniformano il trattamento delle risposte HTTP non valide.
\end{itemize}

\paragraph{Principi SOLID rispettati}
La Facade favorisce il \textbf{Single Responsibility Principle}: i componenti UI gestiscono presentazione e stato locale, mentre il layer servizi centralizza la comunicazione remota. L'approccio rispetta anche l'\textbf{Interface Segregation Principle}, esponendo API frontend mirate per casi d'uso specifici.

\noindent\textbf{Caso d'uso (\texttt{cartService}):} operazioni come \texttt{addItem}, \texttt{removeItem} e \texttt{updateQty} coordinano routing endpoint, serializzazione parametri (es. \texttt{cod\_art}, \texttt{quantity}) e invocazione rete. L'incapsulamento riduce la superficie di modifica: variazioni degli endpoint o dello schema di risposta sono localizzate nel solo Service Layer.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.78\textwidth]{componenti_specifiche_tecniche/architetturalogica/facade_uml.png}
    \caption{Pattern Facade nel frontend: \texttt{cartService} come interfaccia semplificata tra UI e client API.}
    \label{fig:facade-architettura-logica}
\end{figure}

\paragraph{Observer (SSE broadcaster + observer client)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Gli aggiornamenti real-time di sessione (es. carrello) devono essere propagati a piu' consumer in modo coerente e senza polling.
    \item \textbf{Soluzione Architetturale:} Il subject e' \texttt{SSEBroadcaster}, che mantiene i subscriber per sessione; gli observer implementano \texttt{IObserver} e ricevono eventi tramite \texttt{update(event)}.
    \item \textbf{Realizzazione:} L'implementazione adottata e' unica e coincide con il diagramma: \texttt{CartService} emette eventi verso \texttt{SSEBroadcaster}, mentre \texttt{SSEClientQueue} materializza la coda asincrona del singolo subscriber SSE.
\end{itemize}

\paragraph{Principi SOLID rispettati}
Il pattern rispetta l'\textbf{Open/Closed Principle}: nuovi observer possono essere introdotti senza modificare il subject. Inoltre supporta il \textbf{Dependency Inversion Principle}, perche' il broadcaster interagisce tramite contratto \texttt{IObserver} anziche' dipendere da consumer concreti.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.78\textwidth]{componenti_specifiche_tecniche/architetturalogica/observer_uml.png}
    \caption{Pattern Observer per propagazione eventi real-time via SSE.}
    \label{fig:observer-architettura-logica}
\end{figure}

\paragraph{Singleton (module-level services)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Le chiamate API devono mantenere configurazione uniforme ed evitare duplicazioni di client e logica di rete.
    \item \textbf{Soluzione Architetturale:} I servizi sono esportati come istanze condivise a livello modulo, con configurazione centralizzata del client HTTP.
    \item \textbf{Realizzazione:} L'interfaccia utilizza una singola fonte di configurazione per autenticazione, gestione errori e politiche di chiamata, migliorando coerenza e manutenibilita'.
\end{itemize}

\paragraph{Principi SOLID rispettati}
L'uso controllato del Singleton a livello modulo mantiene una responsabilita' unica di configurazione e ciclo vita del client (\textbf{Single Responsibility Principle}). I componenti consumatori restano disaccoppiati dal dettaglio di inizializzazione e dipendono da un servizio gia' configurato, coerente con il \textbf{Dependency Inversion Principle} applicativo.

\noindent\textbf{Caso d'uso (\texttt{cartService} + \texttt{CartProvider}):} il modulo \texttt{cartService} e' esportato come istanza condivisa e centralizza la comunicazione HTTP con il backend. In parallelo, \texttt{CartProvider} gestisce l'unica sorgente di verita' dello stato carrello per la sessione utente: la funzione \texttt{updateQty} applica aggiornamento ottimistico locale, annulla eventuali timer gia' attivi per lo stesso prodotto e persiste la modifica dopo una finestra di debounce di 600 ms. In caso di errore remoto, il provider riallinea lo stato ricaricando i dati reali dal server.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.78\textwidth]{componenti_specifiche_tecniche/architetturalogica/singleton_uml.png}
    \caption{Pattern Singleton nel frontend: \texttt{cartService} come servizio condiviso e \texttt{CartProvider} come sorgente unica dello stato carrello.}
    \label{fig:singleton-architettura-logica}
\end{figure}

\paragraph{Strategie applicative: Optimistic UI e Debounce}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Le operazioni frequenti sul carrello devono restare reattive lato utente, ma senza generare picchi di richieste concorrenti verso il backend.
    \item \textbf{Soluzione Architetturale:} Lo stato locale viene aggiornato in modo ottimistico e le chiamate ripetute in rapida sequenza vengono aggregate tramite debounce.
    \item \textbf{Realizzazione:} Le modifiche sono visibili subito in UI; in caso di errore viene eseguito rollback/sincronizzazione, mentre gli update ravvicinati vengono consolidati in una richiesta finale.
\end{itemize}

\section{Database}

Questa sezione descrive l'architettura dei dati del sistema \textit{SmartOrder}, definendo il ruolo del database, le macro-aree informative gestite, i vincoli di integrità applicati e le modalità di interscambio dati con il sistema gestionale esterno.

\subsection{Ruolo del database nel sistema}
Il sistema si avvale di \textbf{PostgreSQL} (in hosting cloud su Railway) come unica fonte dati transazionale (Single Source of Truth) per il backend FastAPI. 
Il database relazionale assolve a tre ruoli fondamentali:
\begin{itemize}
    \item \textbf{Persistenza Transazionale:} memorizza in modo consistente lo stato delle sessioni, dei carrelli in lavorazione, dei ticket di assistenza e lo storico degli ordini consolidati.
    \item \textbf{Replica del Catalogo ERP:} ospita le tabelle anagrafiche dei prodotti e dei clienti sincronizzate dal gestionale aziendale, permettendo al sistema di operare con logiche di business locali (es. verifica assortimenti).
    \item \textbf{Motore di Ricerca Vettoriale:} tramite l'estensione \texttt{pgvector}, il database archivia gli \textit{embeddings} generati dai modelli LLM e risolve le query semantiche di "Vector Search", mappando le descrizioni in linguaggio naturale ai codici articolo reali.
\end{itemize}

\subsection{Struttura delle principali tabelle}
In base allo schema SQL in uso, di seguito è riportata la struttura e il significato dei campi più rilevanti per le entità centrali del sistema.

\paragraph{Anagrafica Articoli (\texttt{anaart})}
Tabella fondamentale per la gestione del catalogo ordini:
\begin{itemize}
    \item \texttt{cod\_art}: codice articolo (Primary Key).
    \item \texttt{des\_art}: descrizione estesa del prodotto.
    \item \texttt{stato}: stato logico di disponibilità del prodotto.
    \item \texttt{linea}: indica il settore reale a cui appartiene il prodotto (es. acqua, vino, ecc.).
    \item \texttt{famiglia}: indica la fascia o qualità del prodotto (es. premium, standard, ecc.).
    \item \texttt{des\_um}: unità di misura base di vendita (es. casse, cartone).
    \item \texttt{des\_tipo\_um}: unità di misura dei singoli prodotti all'interno della singola confezione (es. pezzi, bottiglie).
    \item \texttt{pezzi\_conf}: quantità calcolata in \texttt{des\_tipo\_um} situata all'interno della misura base \texttt{des\_um}.
    \item \texttt{peso\_netto\_conf}: peso netto per singola unità di \texttt{des\_um}.
\end{itemize}
\textit{Nota operativa:} L'attributo nativo \texttt{settore} presente a schema non contiene in realtà i valori di business indicanti il settore del prodotto, i quali si trovano invece nel campo \texttt{linea}.

\paragraph{Sessioni e Chat (\texttt{chat\_sessions}, \texttt{chat\_messages})}
Il nucleo conversazionale salva lo stato delle interazioni in maniera persistente:
\begin{itemize}
    \item \texttt{status}: indica se la sessione è \textit{active}, \textit{completed} o \textit{archived}.
    \item \texttt{sender} (\texttt{chat\_messages}): traccia l'emittente esatto del messaggio tra \textit{user}, \textit{ai}, \textit{operator} e \textit{system}.
    \item \texttt{image\_data} e \texttt{ocr\_text}: permettono l'invio di ordini multimodali memorizzando un'immagine decodificata in base64 e l'interpretazione testuale del servizio LLM associato.
\end{itemize}

\paragraph{Carrello (\texttt{cart\_items})}
Le righe del carrello in elaborazione prevedono una cronologia di modifica e confidenza:
\begin{itemize}
    \item \texttt{cod\_art} e \texttt{qta}: riferimento all'articolo in assortimento e alle quantità richieste dall'utente.
    \item \texttt{source} e \texttt{last\_updated\_by}: attribuiscono l'inserimento/modifica iniziale e finale a un utente, all'intelligenza artificiale o a un operatore.
    \item \texttt{ai\_confidence}: livello di certezza estrapolato dal LLM (fondamentale per far scattare controlli automatizzati o trigger per il backoffice in caso di bassi valori).
\end{itemize}

\paragraph{Ordini Finalizzati (\texttt{orders}, \texttt{order\_items})}
Rappresentano la formalizzazione del carrello al netto delle modifiche di \textit{Human-in-the-Loop}. Gli ordini mantengono un tracciamento dei campi \texttt{source} per finalità di audit. Troviamo inoltre il flag di stato \texttt{esportato} per gestire l'avvenuta sincronizzazione JSON.

\paragraph{Identità Temporanee e Persistenti (\texttt{app\_users}, \texttt{anacli}, \texttt{asscli})}
\begin{itemize}
    \item \texttt{app\_users}: include il ruolo autenticato (\textit{customer} o \textit{admin}), il collegamento al cliente via \texttt{cod\_cli}, e soprattutto il campo \texttt{export\_folder} che specifica il path assoluto della cartella di rete dove l'operatore desidera ricevere i file JSON degli ordini finalizzati. Questo path è configurabile per operatore ed è fondamentale per l'integrazione ERP distribuita.
    \item \texttt{anacli}: generalità e ragione sociale del cliente.
    \item \texttt{asscli}: tabella ponte di assortimento (collega \texttt{cod\_cli} e \texttt{cod\_art}). Esclusivamente quanto è qui presente risulta ordinabile dal Cliente interessato.
\end{itemize}

\paragraph{Gestione Ticket e Lock Operatore (\texttt{tickets})}
Gestisce la supervisione manuale di casi a bassa confidence AI o anomalie:
\begin{itemize}
    \item \texttt{status}: stati discreti ("aperto", "in\_lavorazione", "chiuso") con vincoli \textit{CHECK}.
    \item \texttt{locked\_by}: ID dell'operatore che ha preso in carico il ticket. Implementa un meccanismo di pessimistic locking per prevenire race condition: un operatore non può modificare un ticket già bloccato da un altro operatore.
    \item \texttt{created\_at}, \texttt{updated\_at}: timestamp per audit trail e metriche di SLA.
\end{itemize}

\paragraph{Feedback e Ottimizzazione LLM (\texttt{message\_feedbacks})}
Registra il feedback implicito ed esplicito sui messaggi AI generati:
\begin{itemize}
    \item \texttt{is\_positive}: valore booleano che indica se l'utente ha trovato utile il messaggio.
    \item \texttt{reason\_category}: categorizzazione del feedback (es. "errato", "incomplete", "inappropriate").
    \item \texttt{comment}: note testuali dell'utente. Questi dati alimentano il processo di retraining e fine-tuning del modello LLM.
    \item Vincolo \textit{UNIQUE}(\texttt{message\_id}, \texttt{user\_id}): ogni utente può fornire un solo feedback per messaggio, evitando duplicati.
\end{itemize}

\paragraph{Embedding Vettoriali per Ricerca Semantica (\texttt{product\_embeddings})}
Archivia gli embedding generati dai modelli LLM per abilitare la vector search:
\begin{itemize}
    \item \texttt{embedding}: vettore multidimensionale memorizzato tramite l'estensione PostgreSQL \texttt{pgvector}. Ciascun embedding rappresenta la "semantica" testuale dell'articolo (descrizione, linea, famiglia).
    \item \texttt{created\_at}: timestamp di generazione dell'embedding (utile per invalidazione e rigenerazione periodica).
    \item Utilizzo: quando l'utente descrive un prodotto in linguaggio naturale, il sistema genera un embedding della query e cerca i \textit{nearest neighbors} nello spazio vettoriale per recuperare articoli semanticamente simili, anche se con descrizioni non letteralmente corrispondenti.
\end{itemize}

\subsection{Uso dei log per analisi operativa e miglioramento}
I dati persistiti nel database non sono utilizzati solo per finalità transazionali, ma potranno anche essere usate per monitoraggio, audit e miglioramento continuo del sistema:
\begin{itemize}
    \item \textbf{Cronologia messaggi (\texttt{chat\_messages})}: consente di ricostruire il contesto conversazionale end-to-end, fondamentale per comprendere richieste ellittiche e analizzare casi anomali in fase di debug.
    \item \textbf{Metadati di confidenza (\texttt{metadata} dei messaggi AI)}: permettono di valutare come il modello ha selezionato articoli e con quale affidabilita', supportando analisi causale degli errori di suggerimento.
    \item \textbf{Feedback utenti (\texttt{message\_feedbacks})}: rendono misurabile la qualita' percepita delle risposte AI e costituiscono un dataset supervisionato per prioritizzare ottimizzazioni future (es. RLHF).
    \item \textbf{Stato carrello e ordini (\texttt{cart\_items}, \texttt{orders}, \texttt{order\_items})}: abilita audit trail operativo su chi ha modificato cosa e quando, utile per verifica di conformità.
    \item \textbf{Ticket di intervento (\texttt{tickets})}: forniscono indicatori sui limiti dell'automazione (frequenza takeover, tempi di gestione), utili per orientare backlog tecnico e miglioramenti di prodotto.
\end{itemize}

In prospettiva, la combinazione di questi log permette di definire KPI tecnici e di processo (precisione suggerimenti, tempo medio takeover, tasso correzioni manuali) utili per guidare decisioni di evoluzione architetturale.

\subsection{Strategie di indicizzazione e performance}
Per garantire responsiveness della ricerca e delle operazioni transazionali, il schema implementa indici compositi e strategici:
\begin{itemize}
    \item \textbf{Indici Compositi su Sessioni e Chat}: \texttt{idx\_chat\_sessions\_user\_status} (user\_id, status) accelera il retrieval delle sessioni attive di un utente. \texttt{idx\_chat\_messages\_session\_id} ottimizza il caricamento della cronologia.
    \item \textbf{Indici su Feedback}: \texttt{idx\_message\_feedbacks\_message\_user\_created} permette query efficienti sul feedback per singolo messaggio, utente e timestamp, fondamentali per il report dei contributi.
    \item \textbf{Vincolo Unico Composito}: \texttt{uq\_cart\_session\_product} previene inserimento duplicato di uno stesso articolo in un carrello nella medesima sessione.
    \item \textbf{Indice Vettoriale}: \texttt{idx\_product\_embeddings\_cosine} utilizza l'algoritmo IVFFlat (Inverted File Flat) con 100 liste per la ricerca vettoriale efficiente su high-dimensional space. L'operatore di distanza è la "cosine distance", appropriato per embedding semantici normalizzati.
\end{itemize}

\subsection{Vincoli dati rilevanti}
Per garantire la robustezza del sistema e prevenire l'invio di ordini commercialmente invalidi al sistema ERP, il database impone rigorosi vincoli di integrità e di dominio:
\begin{itemize}
    \item \textbf{Vincolo di Assortimento (Foreign Key logica):} un prodotto può essere inserito nel carrello di una sessione solo se esiste una corrispondenza valida nella tabella degli assortimenti autorizzati (\texttt{asscli}) per il Cliente in esame.
    \item \textbf{Vincolo di Disponibilità:} controlli a livello database (\texttt{anaart}) impediscono l'immissione in ordine di articoli marcati con stato "sospeso" o "non disponibile".
    \item \textbf{Integrità Referenziale Standard:} chiavi esterne (Foreign Keys) con regole di propagazione \textit{ON DELETE CASCADE/RICT} collegano indissolubilmente le righe dell'ordine all'ID Ordine padre e alla relativa anagrafica Cliente, prevenendo record orfani.
    \item \textbf{Vincolo di Quantità Commerciale:} i campi quantitativi delle righe ordine possiedono vincoli (\textit{CHECK constraints}) per impedire valori nulli, negativi o incompatibili con i multipli di vendita (es. logica a "casse").
\end{itemize}

\subsection{Output verso ERP}
Il database di SmartOrder funge da ambiente di \textit{staging} e composizione. Al termine del processo di checkout, i dati transazionali (intestazione ordine, righe, quantità e ID cliente) estratti dalle tabelle relazionali subiscono un processo di serializzazione. 

Il sistema genera un \textbf{payload strutturato in formato JSON}. Questo file viene immediatamente depositato in una cartella condivisa nel computer, che rappresenta il reale punto di aggancio e consegna formale verso il sistema ERP aziendale.

\subsubsection{Struttura del payload JSON esportato}
L'output e' un oggetto completo che include testata ordine, dettaglio righe e contesto conversazionale. Una forma semplificata del tracciato e' la seguente:

\begin{lstlisting}[language=json,caption={Struttura logica del JSON esportato verso ERP.},label={lst:json-export-erp}]
{
    "order_id": "ORD-2026-000123",
    "cod_cli": "CLI0456",
    "data_ord": "2026-04-14T10:22:31Z",
    "session_id": "sess_7f8b9c",
    "items": [
        {
            "cod_art": "A001",
            "qta_ordinata": 4,
            "des_art": "Acqua Naturale 1L",
            "des_um": "casse",
            "linea": "beverage",
            "famiglia": "standard",
            "source": "ai",
            "ai_confidence": 0.93
        }
    ],
    "messages": [
        {
            "sender": "user",
            "content": "Aggiungi altre 2 casse",
            "feedback": null
        }
    ]
}
\end{lstlisting}

\noindent\textbf{Campi principali:}
\begin{itemize}
        \item \textbf{Testata ordine}: \texttt{order\_id}, \texttt{cod\_cli}, \texttt{data\_ord}, \texttt{session\_id} identificano in modo univoco documento, cliente e contesto operativo.
        \item \textbf{Dettaglio articoli (\texttt{items})}: ogni riga include identificativi ERP (\texttt{cod\_art}), quantita' (\texttt{qta\_ordinata}), metadati descrittivi e attributi di provenienza/affidabilita' (\texttt{source}, \texttt{ai\_confidence}).
        \item \textbf{Contesto conversazionale (\texttt{messages})}: storico dei messaggi e dei feedback associati, utile per tracciabilita' delle decisioni e verifica ex-post.
\end{itemize}

\noindent\textbf{Motivazioni architetturali (Enterprise Ready):}
\begin{itemize}
        \item \textbf{Integrazione standard}: JSON e' interoperabile con ERP moderni e processi ETL/ELT senza conversioni proprietarie.
        \item \textbf{Mappatura diretta}: l'uso di \texttt{cod\_cli} e \texttt{cod\_art} consente caricamento deterministico verso anagrafiche gestionali esistenti.
        \item \textbf{Automazione batch}: la scrittura in cartella condivisa permette import incrementale da parte del gestionale senza inserimento manuale.
        \item \textbf{Audit e compliance}: campi \texttt{source}, \texttt{ai\_confidence} e \texttt{messages} rendono spiegabile la formazione dell'ordine.
        \item \textbf{Separazione delle responsabilita'}: SmartOrder normalizza e valida il dato; l'ERP mantiene i processi amministrativi e logistici a valle.
\end{itemize}

\textit{Nota architetturale - Disaccoppiamento tramite Adapter:} In conformità al pattern Ports and Adapters, il Core applicativo di SmartOrder \textbf{non scrive mai direttamente} nel database dell'ERP. Invece, i servizi di dominio delegano la responsabilità dell'output a un \textbf{driven adapter specializzato} (Export Adapter). Questo adapter:
\begin{itemize}
    \item Riceve dal Core l'ordine strutturato come entità di dominio (pura logica di business).
    \item Trasforma l'entità in un payload JSON conforme al formato atteso dal sistema gestionale.
    \item Scrive il file nella cartella condivisa del File System.
\end{itemize}
Questo disaccoppiamento consente evoluzione indipendente: se domani il sistema ERP cambia formato o destinazione (es. database diretti, API , message queue), basterà sostituire o estendere l'adapter senza toccare la logica di dominio o i servizi di business.
