\section{Introduzione}

\subsection{Scopo del documento}
Il presente documento ha lo scopo di descrivere l'architettura software del sistema \textit{SmartOrder}, definendo le scelte tecnologiche, i pattern architetturali e progettuali adottati, e l'architettura di dettaglio di ciascun componente.
La Specifica Tecnica costituisce il riferimento principale per la fase di codifica, garantendo coerenza tra progettazione e implementazione.

\subsection{Scopo del prodotto}
\textit{SmartOrder} \`e una piattaforma B2B per l'automazione della gestione degli ordini commerciali. Il sistema consente ai clienti di formulare ordini in linguaggio naturale (testo, audio, immagini) e li converte automaticamente in ordini strutturati pronti per l'integrazione con sistemi ERP aziendali, riducendo errori e tempi di inserimento manuale.

\subsection{Avvio manuale (senza Docker)}
% -------------------------------------------------------------------

La procedura di avvio descritta in questa sottosezione ha lo scopo di consentire l'esecuzione locale del sistema per attivit\`a di verifica tecnica, debug e dimostrazione. Essa non rappresenta la procedura di rilascio in ambiente di produzione, che \`e invece gestita tramite pipeline di deploy automatizzato sulla piattaforma Railway.

Questa prevede l'utilizzo di due terminali separati, uno per il backend e uno per il frontend.

\subsubsection{Terminale 1 - Backend (FastAPI)}
\begin{enumerate}
    \item Posizionarsi nella cartella del backend:
          \begin{lstlisting}[language=bash]
cd backend
\end{lstlisting}
    \item Creare l'ambiente virtuale Python:
          \begin{lstlisting}[language=bash]
python3.11 -m venv venv
\end{lstlisting}
    \item Attivare l'ambiente virtuale:
          \begin{lstlisting}[language=bash]
source venv/bin/activate
# Windows: venv\Scripts\activate
\end{lstlisting}
    \item Installare le dipendenze:
          \begin{lstlisting}[language=bash]
pip install -r requirements.txt
\end{lstlisting}
    \item Avviare il backend:
          \begin{lstlisting}[language=bash]
python main.py
\end{lstlisting}
\end{enumerate}

Lasciare questo terminale in esecuzione e aprirne un secondo per il frontend.

\subsubsection{Terminale 2 - Frontend (Next.js)}
\begin{enumerate}
    \item Installare le dipendenze del frontend:
          \begin{lstlisting}[language=bash]
npm install
\end{lstlisting}
    \item Avviare il server di sviluppo:
          \begin{lstlisting}[language=bash]
npm run dev
\end{lstlisting}
\end{enumerate}

Anche questo terminale deve rimanere in esecuzione durante l'utilizzo dell'applicazione.

\subsubsection{Accesso alla webapp}
La webapp \`e disponibile all'indirizzo \texttt{http://localhost:3000}.
In ambiente di sviluppo, le modifiche al codice vengono applicate in tempo reale tramite aggiornamento live.

% -------------------------------------------------------------------
\subsection{Deployment in produzione (Railway)}
% -------------------------------------------------------------------

Il deployment in produzione del sistema \textit{SmartOrder} \`e gestito tramite la piattaforma \textbf{Railway}, utilizzata come ambiente di hosting per l'applicazione web e per il database PostgreSQL.

In sintesi, il flusso di rilascio \`e il seguente:
\begin{itemize}
    \item il codice sorgente viene mantenuto sul repository GitHub del progetto;
    \item Railway \`e configurato per monitorare il branch principale del repository (ad esempio \texttt{main}) e avviare automaticamente una build ad ogni push;
    \item durante la build, Railway installa le dipendenze necessarie (pacchetti Node.js e Python) e costruisce l'immagine eseguibile dell'applicazione;
    \item al termine della build, Railway attiva i servizi applicativi esponendoli tramite URL pubblico HTTPS;
    \item le variabili d'ambiente di produzione (database, chiavi API, parametri di configurazione) vengono gestite all'interno della console di Railway e non sono versionate nel repository.
\end{itemize}

In ambiente di produzione:
\begin{itemize}
    \item il backend (modulo FastAPI) e il frontend (Next.js) vengono eseguiti come servizi gestiti da Railway, rispettando le risorse e i limiti configurati sul progetto;
    \item il database PostgreSQL \`e ospitato su un'istanza Railway dedicata, raggiungibile tramite i parametri configurati nelle variabili d'ambiente;
    \item la piattaforma gestisce automaticamente aspetti infrastrutturali come il bilanciamento del carico, la terminazione TLS e il riavvio dei servizi in caso di errore.
\end{itemize}

L'URL pubblico dell'istanza di produzione \`e fornito da Railway (ad esempio \texttt{https://xyz.railway.app/}) e costituisce il punto di accesso esterno al sistema per gli utenti finali.

% -------------------------------------------------------------------
\subsection{Linguaggi di programmazione}
% -------------------------------------------------------------------

\subsubsection{TypeScript}
\begin{itemize}[noitemsep]
\item \textbf{Versione:} 5
\item \textbf{Documentazione ufficiale:} \href{https://www.typescriptlang.org/docs/}{TypeScript
 Documentation} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Frontend
\item \textbf{Contesto di utilizzo:} linguaggio principale per l'interfaccia utente dell'applicativo Next.js, sia per i componenti React che per la gestione dello stato lato client.
\item \textbf{Vantaggi:} tipizzazione statica che riduce errori a runtime, ottimo supporto IDE, compatibilit\`a nativa con React e Next.js.
\item \textbf{Svantaggi:} overhead di compilazione rispetto a JavaScript puro.
\end{itemize}

\subsubsection{Python}
\begin{itemize}[noitemsep]
\item \textbf{Versione:} 3.11
\item \textbf{Documentazione ufficiale:} \href{https://docs.python.org/3/}{Python 3 Documentation} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Backend / Modulo AI
\item \textbf{Contesto di utilizzo:} implementazione della pipeline di elaborazione degli ordini tramite intelligenza artificiale e gestione delle API REST tramite FastAPI.
\item \textbf{Vantaggi:} ecosistema maturo in ambito NLP/ML, compatibilit\`a nativa con le librerie OpenAI, sintassi concisa, supporto asincrono con Uvicorn/ASGI.
\item \textbf{Svantaggi:} prestazioni inferiori a linguaggi compilati; mitigato dall'uso di Uvicorn (ASGI asincrono).
\end{itemize}

\subsubsection{SQL}
\begin{itemize}[noitemsep]

\item \textbf{Ambito:} Database
\item \textbf{Contesto di utilizzo:} interrogazioni sul catalogo prodotti, gestione ordini, sessioni e carrelli.
\end{itemize}
