\section{Introduzione}

\subsection{Scopo del documento}
Il presente documento ha lo scopo di descrivere l'architettura software del sistema \textit{SmartOrder}, definendo le scelte tecnologiche, i pattern architetturali e progettuali adottati, e l'architettura di dettaglio di ciascun componente.
La Specifica Tecnica costituisce il riferimento principale per la fase di codifica, garantendo coerenza tra progettazione e implementazione.

\subsection{Scopo del prodotto}
\textit{SmartOrder} \`e una piattaforma B2B per l'automazione della gestione degli ordini commerciali. Il sistema consente ai clienti di formulare ordini in linguaggio naturale (testo, audio, immagini) e li converte automaticamente in ordini strutturati pronti per l'integrazione con sistemi ERP aziendali, riducendo errori e tempi di inserimento manuale.

\subsection{Glossario}
Per evitare ambiguit\`a, i termini tecnici e di dominio utilizzati nel documento sono definiti nel \textit{Glossario v2.0}.
I termini presenti nel glossario sono contrassegnati con il simbolo \textsubscript{\textsc{g}} alla prima occorrenza in ogni sezione.

\section{Tecnologie utilizzate}

In questa sezione vengono elencate le tecnologie adottate per lo sviluppo del sistema \textit{SmartOrder}. Per ciascuna tecnologia viene indicata la versione utilizzata, la motivazione della scelta e il contesto d'impiego nel progetto.

% -------------------------------------------------------------------
\subsection{Linguaggi di programmazione}
% -------------------------------------------------------------------

\subsubsection{TypeScript}
\begin{itemize}[noitemsep]
\item \textbf{Versione:} 5
\item \textbf{Documentazione ufficiale:} \href{https://www.typescriptlang.org/docs/}{TypeScript
 Documentation} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Frontend
\item \textbf{Contesto di utilizzo:} linguaggio principale per l'interfaccia utente dell'applicativo Next.js, sia per i componenti React che per la gestione dello stato lato client.
\item \textbf{Vantaggi:} tipizzazione statica che riduce errori a runtime, ottimo supporto IDE, compatibilit\`a nativa con React e Next.js.
\item \textbf{Svantaggi:} overhead di compilazione rispetto a JavaScript puro.
\end{itemize}

\subsubsection{Python}
\begin{itemize}[noitemsep]
\item \textbf{Versione:} 3.11
\item \textbf{Documentazione ufficiale:} \href{https://docs.python.org/3/}{Python 3 Documentation} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Backend / Modulo AI
\item \textbf{Contesto di utilizzo:} implementazione della pipeline di elaborazione degli ordini tramite intelligenza artificiale e gestione delle API REST tramite FastAPI.
\item \textbf{Vantaggi:} ecosistema maturo in ambito NLP/ML, compatibilit\`a nativa con le librerie OpenAI, sintassi concisa, supporto asincrono con Uvicorn/ASGI.
\item \textbf{Svantaggi:} prestazioni inferiori a linguaggi compilati; mitigato dall'uso di Uvicorn (ASGI asincrono).
\end{itemize}

\subsubsection{SQL}
\begin{itemize}[noitemsep]
\item \textbf{Documentazione ufficiale:} \href{https://www.iso.org/standard/76584.html}{ISO/IEC 9075:2023 (SQL)} (Ultima modifica 2026-03-26)
\item \textbf{Ambito:} Database
\item \textbf{Contesto di utilizzo:} interrogazioni sul catalogo prodotti, gestione ordini, sessioni e carrelli.
\end{itemize}

\section{Architettura di deployment}

Questa sezione descrive la distribuzione fisica dei componenti del sistema \textit{SmartOrder}, illustrando gli ambienti di esecuzione, i nodi infrastrutturali coinvolti e le strategie di Continuous Integration e Continuous Deployment (CI/CD\ped{g}) adottate.

\subsection{Architettura: Modular Monolith}
L'applicazione \textit{SmartOrder} adotta un'architettura \textbf{Modular Monolith}. Il termine riflette due proprietà fondamentali:
\begin{itemize}
    \item \textbf{Monolitica}: l'intera applicazione viene distribuita come un unico processo (il file \texttt{main.py} del backend FastAPI), il quale ospita tutti i servizi e la logica di business.
    \item \textbf{Modulare}: nonostante sia un singolo processo, la logica di business è organizzata e isolata \textbf{verticalmente} per dominio. Ogni modulo attraversa tutti i layer dell'architettura (dal controller alla persistenza).
\end{itemize}

\subsubsection{Separazione logica e disaccoppiamento}
Le principali caratteristiche della struttura modulare sono:
\begin{itemize}
    \item \textbf{Separazione per Dominio}: Ogni dominio applicativo (es. Order, Ticket, Carrello, Sessione) possiede il proprio \textbf{Service} (logica di business) e il proprio \textbf{Controller} (entry point API). I servizi rimangono indipendenti l'uno dall'altro, evitando dipendenze circolari e semplificando la manutenzione.
    
    \item \textbf{Disaccoppiamento dall'Infrastruttura}: I servizi non accedono direttamente al database, ai file storage o ai servizi esterni (OpenAI, ERP). Ogni accesso esterno avviene attraverso \textbf{Ports} (interfacce), che definiscono il contratto necessario. Le implementazioni concrete (Adapter) sono fornite dal framework FastAPI tramite Dependency Injection, rendendo la logica applicativa completamente indipendente dalla tecnologia di persistenza o dall'infrastruttura utilizzata.
\end{itemize}

\subsubsection{Definizione di ``Modulo'' nella nostra App}
Nella architettura di \textit{SmartOrder}, un \textbf{Modulo} è un'unità funzionale autonoma e coesiva che attraversa verticalmente tutti i layer dell'architettura. Un modulo non è una semplice cartella nel filesystem, ma l'insieme coordinato dei seguenti componenti:

\begin{enumerate}
    \item \textbf{Entry Point (Controller)}: Un file controller (es. \texttt{order\_controller.py}) che espone l'interfaccia HTTP del modulo. Definisce gli endpoint REST, valida i parametri ricevuti dal client e delega l'elaborazione al Service.
    
    \item \textbf{Logica di Business (Service)}: Il cuore del modulo (es. \texttt{order\_service.py}). Qui risiede l'intera intelligenza applicativa: le regole di dominio, l'orchestrazione tra moduli, la validazione transazionale e la coordinazione con servizi esterni (es. OpenAI).
    
    \item \textbf{Contratto di Persistenza (Port)}: Un'interfaccia astratta (es. \texttt{i\_order\_repository.py}) che dichiara quali dati il modulo necessita. Il Service dipende da questa interfaccia, non da implementazioni concrete. La Port definisce operazioni come \texttt{save()}, \texttt{find\_by\_id()}, \texttt{delete()}, etc.
    
    \item \textbf{Dati (Schema/Domain)}: Gli oggetti di trasferimento dati del modulo (definiti in \texttt{schemas.py}). Rappresentano la forma dei dati che il modulo scambia con il mondo esterno: strutture in ingresso (Request DTO) e in uscita (Response DTO).
    
    \item \textbf{Implementazione (Adapter)}: Il codice specifico che parla con il database, il file storage o sistemi terzi (es. \texttt{postgres\_adapter.py}). Implementa concretamente la Port, traducendo le chiamate del Service in operazioni SQL, chiamate API REST o I/O su file. Adapter diversi possono implementare la medesima Port, permettendo di sostituire la persistenza senza toccare il Service.
\end{enumerate}

Questa struttura garantisce che il modulo sia \textbf{coeso} (tutti i componenti concorrono a una medesima responsabilità di business), \textbf{disaccoppiato} (il Service non conosce come i dati sono effettivamente memorizzati) e \textbf{testabile} (la logica di business può essere testata in assenza di database reale, mockando la Port).

\subsection{Scenario di esecuzione}
Il sistema prevede due configurazioni principali per l'esecuzione, pensate per separare l'ambiente di sviluppo da quello di produzione:
\begin{itemize}
    \item \textbf{Ambiente di Sviluppo (Locale)}: i servizi di frontend e backend vengono eseguiti localmente sulla macchina di sviluppo, sfruttando la containerizzazione (Docker Compose) o task eseguiti in locale per avviare l'applicativo e un database PostgreSQL isolato. Questo garantisce un ambiente di test riproducibile.

Railway gestisce nativamente il provisioning delle risorse, ospitando in container isolati i servizi di frontend (Next.js), backend (FastAPI) e il database relazionale (PostgreSQL).
\end{itemize}

\begin{itemize}
    \item \textbf{Frontend $\leftrightarrow$ Backend}: le chiamate API avvengono su canale cifrato HTTPS, scambiando payload in formato JSON. L'autenticazione è garantita tramite token o cookie di sessione sicuri.
    \item \textbf{Backend $\leftrightarrow$ PostgreSQL}: connessione su protocollo TCP proprietario del database, ottimizzata tramite un meccanismo di \textit{connection pooling} (gestito tramite \texttt{psycopg2}) per sopportare picchi di traffico.
    \item \textbf{Backend $\leftrightarrow$ OpenAI API}: invocazioni REST su rete pubblica cifrata HTTPS verso i server cloud di OpenAI.
    \item \textbf{Backend $\rightarrow$ File Storage}: scritture nella cartella condivisa tramite operazioni di I/O su filesystem, necessarie a depositare il file JSON dell'ordine per il recupero da parte del sistema ERP.
\end{itemize}

\subsection{Aspetti operativi e CD}
Per garantire rilasci continui, controllati e privi di disservizi, il progetto adotta pratiche automatizzate di \textit{Continuous Deployment} (CD\ped{g}):
\begin{itemize}
    \item \textbf{Continuous Deployment (CD\ped{g}) automatizzato}: il deployment in produzione è delegato a \textbf{Railway}, il quale è interfacciato direttamente al repository GitHub. Ad ogni push sul branch principale, Railway rileva automaticamente i cambiamenti e lancia il processo di build e deploy. Questo avviene in modalità \textit{zero-downtime}, aggiornando i container senza causare interruzioni agli utenti.
    \item \textbf{Gestione dei Segreti}: chiavi API, credenziali del database e secret di cifratura non vengono mai committati nel repository. Sono configurati come variabili d'ambiente (Environment Variables) criptate direttamente all'interno della dashboard del progetto su Railway.
\end{itemize}

\section{Architettura logica}

Questa sezione descrive la struttura logica di \textit{SmartOrder}, separando i due sottosistemi principali (backend e frontend) e i pattern impiegati per garantire modularita', testabilita' ed evoluzione controllata del codice.

\subsection{Backend}

\subsubsection{Esigenze Architetturali}
Il backend deve mantenere disaccoppiata la logica applicativa dalle tecnologie esterne (HTTP, database, servizi AI), in modo da preservare la stabilita' del dominio al variare dell'infrastruttura e semplificare il collaudo dei servizi.

\subsubsection{Soluzione Architetturale}
L'organizzazione segue un'\textbf{Architettura Esagonale (Ports and Adapters)}:
\begin{itemize}
    \item \textbf{Core}: servizi applicativi e regole di dominio.
    \item \textbf{Ports}: interfacce astratte che definiscono i contratti richiesti dal Core.
    \item \textbf{Adapters}: implementazioni concrete dei Ports e adattatori di ingresso HTTP.
\end{itemize}

\noindent\textbf{Vista di insieme:} il diagramma UML esagonale evidenzia la separazione tra controller (ingresso), servizi di dominio (Core), ports applicative e adapter infrastrutturali (PostgreSQL e OpenAI), con composizione delle dipendenze nel punto di bootstrap (\texttt{main.py}).

    \caption{Architettura logica backend secondo il modello esagonale (Ports and Adapters).}
    \label{fig:esagonale-architettura-logica}

In questo contesto sono applicati i seguenti pattern:
\begin{itemize}
    \item \textbf{Dependency Injection}: composizione delle dipendenze nel punto di avvio applicativo e iniezione nei controller/servizi.
    \item \textbf{Repository}: astrazione della persistenza tramite interfacce dedicate ai casi d'uso di dominio.
\end{itemize}

\subsubsection{Realizzazione}
Il Core opera tramite interfacce, senza dipendere direttamente dai dettagli tecnologici. I controller FastAPI fungono da \textit{driving adapters} traducendo le richieste esterne in invocazioni dei servizi applicativi; gli adapter infrastrutturali (database e AI) fungono da \textit{driven adapters} implementando i Ports. Questa struttura consente di evolvere i dettagli di integrazione mantenendo stabile la logica di dominio.

% -------------------------------------------------------------------
\subsection{Frontend}

\subsubsection{Esigenze Architetturali}
Il frontend richiede una separazione netta tra presentazione, stato applicativo e accesso ai dati, con supporto ad aggiornamenti real-time e interazioni frequenti sul carrello senza degradare la reattivita' dell'interfaccia.

\subsubsection{Soluzione Architetturale}
Il frontend adotta una \textbf{Component-Based Architecture} organizzata per responsabilita':
\begin{itemize}
    \item \textbf{app/}: routing, layout e proxy API verso il backend.
    \item \textbf{components/}: componenti presentazionali per dominio funzionale.
    \item \textbf{contexts/}: orchestrazione dello stato globale e logica applicativa condivisa.
    \item \textbf{hooks/}: logica riusabile legata a sottoscrizioni e ciclo di vita.
    \item \textbf{services/}: accesso ai dati e mappatura degli endpoint backend.
\end{itemize}

I pattern principali adottati sono:
\begin{itemize}
    \item \textbf{Facade}: il layer \texttt{services/} incapsula la comunicazione HTTP, isolando i componenti UI dai dettagli di rete.
    \item \textbf{Observer}: Context API e flussi SSE propagano gli aggiornamenti di stato ai componenti sottoscritti.
    \item \textbf{Singleton (module-level)}: i servizi API sono condivisi come istanze di modulo, con configurazione centralizzata.
\end{itemize}

\subsubsection{Realizzazione}
I componenti UI delegano la logica applicativa ai context e la logica di accesso dati ai servizi. Per il carrello sono applicate strategie operative complementari: \textit{Optimistic UI} per aggiornamento immediato dell'interfaccia, \textit{rollback} in caso di errore e \textit{debounce} sugli update rapidi, riducendo richieste ridondanti mantenendo la coerenza con lo stato server.

% -------------------------------------------------------------------
\subsection{Design pattern adottati}

Questa sottosezione mantiene una vista problem-oriented, separando i pattern effettivamente adottati nel backend e nel frontend.

\subsubsection{Pattern backend}

\paragraph{Dependency Injection}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} I servizi applicativi devono essere testabili e sostituibili senza modificare controller e logica di dominio. Creare dipendenze "in linea" nei componenti avrebbe introdotto accoppiamento forte e inizializzazioni duplicate.
    \item \textbf{Soluzione Architetturale:} Le dipendenze vengono composte nel punto di avvio e iniettate nei componenti tramite provider del framework, mantenendo separati il wiring infrastrutturale e il comportamento di dominio.
    \item \textbf{Realizzazione:} Controller e servizi ricevono istanze gia' risolte; in collaudo e test si possono iniettare implementazioni alternative (mock/stub) senza intervenire sul codice applicativo.
\end{itemize}

\noindent\textbf{Esempio:} il composition root in \texttt{main.py} costruisce adapter e broadcaster, quindi inietta le dipendenze nei servizi applicativi (\texttt{CartService}, \texttt{AuthService}, \texttt{ConversationService}). Nei test le dipendenze possono essere sostituite (es. \texttt{FakeSessionManager}) senza modificare la logica di dominio.

\paragraph{Repository}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} La logica di business non deve dipendere da query SQL, transazioni e dettagli di schema, per evitare propagazione delle modifiche infrastrutturali nel Core.
    \item \textbf{Soluzione Architetturale:} L'accesso ai dati e' mediato da interfacce repository orientate ai casi d'uso (lettura/scrittura ordini, sessioni, ticket), cosi' da mantenere stabile il contratto visto dal dominio.
    \item \textbf{Realizzazione:} I servizi lavorano su repository astratti; gli adapter concreti gestiscono SQL, mapping e vincoli di persistenza, preservando il disaccoppiamento tra dominio e DBMS.
\end{itemize}

\noindent\textbf{Esempio:} \texttt{AuthService} dipende da \texttt{IUserRepository} e usa il contratto per login/lookup utenze. L'implementazione puo' essere \texttt{PostgresAdapter} in esercizio o \texttt{FakeUserRepository} in test: il servizio mantiene invariata l'interfaccia d'uso e resta indipendente dalla persistenza.

\paragraph{Adapter (Ports and Adapters)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Il Core deve integrare database e servizi AI senza conoscere SDK, payload e protocolli specifici dei provider.
    \item \textbf{Soluzione Architetturale:} Le Ports definiscono contratti stabili; i driven adapter traducono le operazioni di dominio in chiamate tecniche verso le integrazioni esterne.
    \item \textbf{Realizzazione:} Nel caso AI, \texttt{ConversationService} dipende da \texttt{IAIClient} e non dall'SDK OpenAI. \texttt{OpenAIAdapter} implementa l'interfaccia e converte i metodi applicativi (\texttt{interpret\_intent}, \texttt{transcribe\_audio}, \texttt{generate\_embedding}) nelle chiamate specifiche di \texttt{AsyncOpenAI}.
\end{itemize}

\noindent\textbf{Esempio:} il diagramma mostra un adattamento \textit{port-to-sdk}: il contratto applicativo resta stabile nel Core, mentre la traduzione dei payload e delle API del provider e' confinata nell'adapter. In questo modo, un cambio di provider AI impatta il layer di integrazione senza modificare il servizio di dominio.

\subsubsection{Pattern frontend}

\paragraph{Facade (Service Layer frontend)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} La UI deve restare disaccoppiata da protocollo HTTP, costruzione URL, header di sicurezza, cookie CSRF e gestione degli errori applicativi.
    \item \textbf{Soluzione Architetturale:} Il Service Layer in \texttt{services/} espone servizi specialistici (es. \texttt{cartService}, \texttt{authService}, \texttt{orderService}) che agiscono da Facade verso le API backend.
    \item \textbf{Realizzazione:} I componenti React invocano operazioni di dominio ad alto livello; internamente i servizi orchestrano utilita' di basso livello (\texttt{apiFetch}, \texttt{buildApiUrl}, \texttt{getApiBaseUrl}, \texttt{getApiErrorMessage}) e uniformano il trattamento delle risposte HTTP non valide.
\end{itemize}

\noindent\textbf{Caso d'uso (\texttt{cartService}):} operazioni come \texttt{addItem}, \texttt{removeItem} e \texttt{updateQty} coordinano routing endpoint, serializzazione parametri (es. \texttt{cod\_art}, \texttt{quantity}) e invocazione rete. L'incapsulamento riduce la superficie di modifica: variazioni degli endpoint o dello schema di risposta sono localizzate nel solo Service Layer.

    \label{fig:facade-architettura-logica}
\end{figure}

\paragraph{Observer (SSE broadcaster + observer client)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Gli aggiornamenti real-time di sessione (es. carrello) devono essere propagati a piu' consumer in modo coerente e senza polling.
    \item \textbf{Soluzione Architetturale:} Il subject e' \texttt{SSEBroadcaster}, che mantiene i subscriber per sessione; gli observer implementano \texttt{IObserver} e ricevono eventi tramite \texttt{update(event)}.
    \item \textbf{Realizzazione:} L'implementazione adottata e' unica e coincide con il diagramma: \texttt{CartService} emette eventi verso \texttt{SSEBroadcaster}, mentre \texttt{SSEClientQueue} materializza la coda asincrona del singolo subscriber SSE.
\end{itemize}

\end{figure}

\paragraph{Singleton (module-level services)}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Le chiamate API devono mantenere configurazione uniforme ed evitare duplicazioni di client e logica di rete.
    \item \textbf{Soluzione Architetturale:} I servizi sono esportati come istanze condivise a livello modulo, con configurazione centralizzata del client HTTP.
    \item \textbf{Realizzazione:} L'interfaccia utilizza una singola fonte di configurazione per autenticazione, gestione errori e politiche di chiamata, migliorando coerenza e manutenibilita'.
\end{itemize}

\paragraph{Strategie applicative: Optimistic UI e Debounce}
\begin{itemize}
    \item \textbf{Esigenze Architetturali:} Le operazioni frequenti sul carrello devono restare reattive lato utente, ma senza generare picchi di richieste concorrenti verso il backend.
    \item \textbf{Soluzione Architetturale:} Lo stato locale viene aggiornato in modo ottimistico e le chiamate ripetute in rapida sequenza vengono aggregate tramite debounce.
    \item \textbf{Realizzazione:} Le modifiche sono visibili subito in UI; in caso di errore viene eseguito rollback/sincronizzazione, mentre gli update ravvicinati vengono consolidati in una richiesta finale.
\end{itemize}

\section{Database}

Questa sezione descrive l'architettura dei dati del sistema \textit{SmartOrder}, definendo il ruolo del database, le macro-aree informative gestite, i vincoli di integrità applicati e le modalità di interscambio dati con il sistema gestionale esterno.

\subsection{Ruolo del database nel sistema}
Il sistema si avvale di \textbf{PostgreSQL} (in hosting cloud su Railway) come unica fonte dati transazionale (Single Source of Truth) per il backend FastAPI. 
Il database relazionale assolve a tre ruoli fondamentali:
\begin{itemize}
    \item \textbf{Persistenza Transazionale:} memorizza in modo consistente lo stato delle sessioni, dei carrelli in lavorazione, dei ticket di assistenza e lo storico degli ordini consolidati.
    \item \textbf{Replica del Catalogo ERP:} ospita le tabelle anagrafiche dei prodotti e dei clienti sincronizzate dal gestionale aziendale, permettendo al sistema di operare con logiche di business locali (es. verifica assortimenti).
    \item \textbf{Motore di Ricerca Vettoriale:} tramite l'estensione \texttt{pgvector}, il database archivia gli \textit{embeddings} generati dai modelli LLM e risolve le query semantiche di "Vector Search", mappando le descrizioni in linguaggio naturale ai codici articolo reali.
\end{itemize}

\subsection{Macro-aree informative}
Il modello relazionale del database è logicamente suddiviso in macro-aree, ciascuna responsabile di un dominio specifico dell'applicazione.

\subsection{Struttura delle principali tabelle}
In base allo schema SQL in uso, di seguito è riportata la struttura e il significato dei campi più rilevanti per le entità centrali del sistema.

\paragraph{Anagrafica Articoli (\texttt{anaart})}
Tabella fondamentale per la gestione del catalogo ordini:
\begin{itemize}
    \item \texttt{cod\_art}: codice articolo (Primary Key).
    \item \texttt{des\_art}: descrizione estesa del prodotto.
    \item \texttt{stato}: stato logico di disponibilità del prodotto.
    \item \texttt{linea}: indica il settore reale a cui appartiene il prodotto (es. acqua, vino, ecc.).
    \item \texttt{famiglia}: indica la fascia o qualità del prodotto (es. premium, standard, ecc.).
    \item \texttt{des\_um}: unità di misura base di vendita (es. casse, cartone).
    \item \texttt{des\_tipo\_um}: unità di misura dei singoli prodotti all'interno della singola confezione (es. pezzi, bottiglie).
    \item \texttt{pezzi\_conf}: quantità calcolata in \texttt{des\_tipo\_um} situata all'interno della misura base \texttt{des\_um}.
    \item \texttt{peso\_netto\_conf}: peso netto per singola unità di \texttt{des\_um}.
\end{itemize}
\textit{Nota operativa:} L'attributo nativo \texttt{settore} presente a schema non contiene in realtà i valori di business indicanti il settore del prodotto, i quali si trovano invece nel campo \texttt{linea}.

\paragraph{Sessioni e Chat (\texttt{chat\_sessions}, \texttt{chat\_messages})}
Il nucleo conversazionale salva lo stato delle interazioni in maniera persistente:
\begin{itemize}
    \item \texttt{status}: indica se la sessione è \textit{active}, \textit{completed} o \textit{archived}.
    \item \texttt{sender} (\texttt{chat\_messages}): traccia l'emittente esatto del messaggio tra \textit{user}, \textit{ai}, \textit{operator} e \textit{system}.
    \item \texttt{image\_data} e \texttt{ocr\_text}: permettono l'invio di ordini multimodali memorizzando un'immagine decodificata in base64 e l'interpretazione testuale del servizio LLM associato.
\end{itemize}

\paragraph{Carrello (\texttt{cart\_items})}
Le righe del carrello in elaborazione prevedono una cronologia di modifica e confidenza:
\begin{itemize}
    \item \texttt{cod\_art} e \texttt{qta}: riferimento all'articolo in assortimento e alle quantità richieste dall'utente.
    \item \texttt{source} e \texttt{last\_updated\_by}: attribuiscono l'inserimento/modifica iniziale e finale a un utente, all'intelligenza artificiale o a un operatore.
    \item \texttt{ai\_confidence}: livello di certezza estrapolato dal LLM (fondamentale per far scattare controlli automatizzati o trigger per il backoffice in caso di bassi valori).
\end{itemize}

\paragraph{Ordini Finalizzati (\texttt{orders}, \texttt{order\_items})}
Rappresentano la formalizzazione del carrello al netto delle modifiche di \textit{Human-in-the-Loop}. Gli ordini mantengono un tracciamento dei campi \texttt{source} per finalità di audit. Troviamo inoltre il flag di stato \texttt{esportato} per gestire l'avvenuta sincronizzazione JSON.

\paragraph{Identità Temporanee e Persistenti (\texttt{app\_users}, \texttt{anacli}, \texttt{asscli})}
\begin{itemize}
    \item \texttt{app\_users}: include il ruolo autenticato (\textit{customer} o \textit{admin}), il collegamento al cliente via \texttt{cod\_cli}, e soprattutto il campo \texttt{export\_folder} che specifica il path assoluto della cartella di rete dove l'operatore desidera ricevere i file JSON degli ordini finalizzati. Questo path è configurabile per operatore ed è fondamentale per l'integrazione ERP distribuita.
    \item \texttt{anacli}: generalità e ragione sociale del cliente.
    \item \texttt{asscli}: tabella ponte di assortimento (collega \texttt{cod\_cli} e \texttt{cod\_art}). Esclusivamente quanto è qui presente risulta ordinabile dal Cliente interessato.
\end{itemize}

\paragraph{Gestione Ticket e Lock Operatore (\texttt{tickets})}
Gestisce la supervisione manuale di casi a bassa confidence AI o anomalie:
\begin{itemize}
    \item \texttt{status}: stati discreti ("aperto", "in\_lavorazione", "chiuso") con vincoli \textit{CHECK}.
    \item \texttt{locked\_by}: ID dell'operatore che ha preso in carico il ticket. Implementa un meccanismo di pessimistic locking per prevenire race condition: un operatore non può modificare un ticket già bloccato da un altro operatore.
    \item \texttt{created\_at}, \texttt{updated\_at}: timestamp per audit trail e metriche di SLA.
\end{itemize}

\paragraph{Feedback e Ottimizzazione LLM (\texttt{message\_feedbacks})}
Registra il feedback implicito ed esplicito sui messaggi AI generati:
\begin{itemize}
    \item \texttt{is\_positive}: valore booleano che indica se l'utente ha trovato utile il messaggio.
    \item \texttt{reason\_category}: categorizzazione del feedback (es. "errato", "incomplete", "inappropriate").
    \item \texttt{comment}: note testuali dell'utente. Questi dati alimentano il processo di retraining e fine-tuning del modello LLM.
    \item Vincolo \textit{UNIQUE}(\texttt{message\_id}, \texttt{user\_id}): ogni utente può fornire un solo feedback per messaggio, evitando duplicati.
\end{itemize}

\paragraph{Embedding Vettoriali per Ricerca Semantica (\texttt{product\_embeddings})}
Archivia gli embedding generati dai modelli LLM per abilitare la vector search:
\begin{itemize}
    \item \texttt{embedding}: vettore multidimensionale memorizzato tramite l'estensione PostgreSQL \texttt{pgvector}. Ciascun embedding rappresenta la "semantica" testuale dell'articolo (descrizione, linea, famiglia).
    \item \texttt{created\_at}: timestamp di generazione dell'embedding (utile per invalidazione e rigenerazione periodica).
    \item Utilizzo: quando l'utente descrive un prodotto in linguaggio naturale, il sistema genera un embedding della query e cerca i \textit{nearest neighbors} nello spazio vettoriale per recuperare articoli semanticamente simili, anche se con descrizioni non letteralmente corrispondenti.
\end{itemize}

\subsection{Strategie di indicizzazione e performance}
Per garantire responsiveness della ricerca e delle operazioni transazionali, il schema implementa indici compositi e strategici:
\begin{itemize}
    \item \textbf{Indici Compositi su Sessioni e Chat}: \texttt{idx\_chat\_sessions\_user\_status} (user\_id, status) accelera il retrieval delle sessioni attive di un utente. \texttt{idx\_chat\_messages\_session\_id} ottimizza il caricamento della cronologia.
    \item \textbf{Indici su Feedback}: \texttt{idx\_message\_feedbacks\_message\_user\_created} permette query efficienti sul feedback per singolo messaggio, utente e timestamp, fondamentali per il report dei contributi.
    \item \textbf{Vincolo Unico Composito}: \texttt{uq\_cart\_session\_product} previene inserimento duplicato di uno stesso articolo in un carrello nella medesima sessione.
    \item \textbf{Indice Vettoriale}: \texttt{idx\_product\_embeddings\_cosine} utilizza l'algoritmo IVFFlat (Inverted File Flat) con 100 liste per la ricerca vettoriale efficiente su high-dimensional space. L'operatore di distanza è la "cosine distance", appropriato per embedding semantici normalizzati.
\end{itemize}

\subsection{Vincoli dati rilevanti}
Per garantire la robustezza del sistema e prevenire l'invio di ordini commercialmente invalidi al sistema ERP, il database impone rigorosi vincoli di integrità e di dominio:
\begin{itemize}
    \item \textbf{Vincolo di Assortimento (Foreign Key logica):} un prodotto può essere inserito nel carrello di una sessione solo se esiste una corrispondenza valida nella tabella degli assortimenti autorizzati (\texttt{asscli}) per il Cliente in esame.
    \item \textbf{Vincolo di Disponibilità:} controlli a livello database (\texttt{anaart}) impediscono l'immissione in ordine di articoli marcati con stato "sospeso" o "non disponibile".
    \item \textbf{Integrità Referenziale Standard:} chiavi esterne (Foreign Keys) con regole di propagazione \textit{ON DELETE CASCADE/RESTRICT} collegano indissolubilmente le righe dell'ordine all'ID Ordine padre e alla relativa anagrafica Cliente, prevenendo record orfani.
    \item \textbf{Vincolo di Quantità Commerciale:} i campi quantitativi delle righe ordine possiedono vincoli (\textit{CHECK constraints}) per impedire valori nulli, negativi o incompatibili con i multipli di vendita (es. logica a "casse").
\end{itemize}

\subsection{Output verso ERP}
Il database di SmartOrder funge da ambiente di \textit{staging} e composizione. Al termine del processo di checkout, i dati transazionali (intestazione ordine, righe, quantità e ID cliente) estratti dalle tabelle relazionali subiscono un processo di serializzazione. 

Il sistema genera un \textbf{payload strutturato in formato JSON}. Questo file viene immediatamente depositato in una cartella condivisa nel computer, che rappresenta il reale punto di aggancio e consegna formale verso il sistema ERP aziendale.

\textit{Nota architetturale - Disaccoppiamento tramite Adapter:} In conformità al pattern Ports and Adapters, il Core applicativo di SmartOrder \textbf{non scrive mai direttamente} nel database dell'ERP. Invece, i servizi di dominio delegano la responsabilità dell'output a un \textbf{driven adapter specializzato} (Export Adapter). Questo adapter:
\begin{itemize}
    \item Riceve dal Core l'ordine strutturato come entità di dominio (pura logica di business).
    \item Trasforma l'entità in un payload JSON conforme al formato atteso dal sistema gestionale.
    \item Scrive il file nella cartella condivisa del File System.
\end{itemize}
Questo disaccoppiamento consente evoluzione indipendente: se domani il sistema ERP cambia formato o destinazione (es. database diretti, API REST, message queue), basterà sostituire o estendere l'adapter senza toccare la logica di dominio o i servizi di business.
